SME Ops System — HR 表單系統完整規格

產出 2026-05-05　·　共 18 種主要表單 + 自訂表單 (form-builder) 機制　·　依「表單中心」分 4 大類整理

核准方式標籤： 自動解析（HR 類，依組織圖找直屬主管） 簽核鏈（依預設步驟逐關推進） v2 新（今日新增） Builder 自訂（用 form-builder 拉欄位）

A. 表單系統架構

HR 表單分兩種架構：

  1. 專屬頁面：每張表單一個 React 頁面（如 [Resignation.jsx](src/pages/hr/Resignation.jsx)）+ 一張 DB 表。適合複雜邏輯與 DB trigger 連動。
  2. 自訂表單（form-builder）：admin 在 /hr/form-builder 拉欄位 → 存 form_templates → 員工填表 → 存 form_submissions。適合輕量級、彈性需求高的表單。

核准方式分兩種：

  • 自動解析：員工送單後，DB function _resolve_hr_approver_ids 找：店長 → 直屬主管 → 部門主管 → 部門樹往上。適用於請假/加班等 HR 類。
  • 簽核鏈：預先在 [/process/approval-chains](/process/approval-chains) 設定固定 N 關步驟，每關指定簽核人/部門/角色。適用於離職/異動/費用報銷等重大決策。

C. 假勤類表單（5 種）

C1. 請假申請

路由/hr/leave
DB 表leave_requests
核准方式自動解析 _resolve_hr_approver_ids
欄位employee_id、leave_type（13 種，見薪資文件 E 區）、start_date、end_date、days、reason、status（待審核/已核准/已駁回/已退回）

備註事假無薪、病假前 30 天半薪、特休全薪。員工餘額不足時前端會擋下

D. 人事異動類表單（5 種）

D1. 離職申請v2 新

路由/hr/forms/resignation
DB 表resignation_requests
核准方式簽核鏈 category='離職'（建議：直屬主管 → HR → 執行長）
欄位employee_id（搜尋下拉）、planned_resign_date、reason（個人/家庭/健康/另謀高就/進修/退休/其他）、reason_detail、handover_notes（交接事項）、approval_chain_id、current_step（0-indexed）
連動最後一關核准 → DB trigger trg_resign_cascade 自動：UPDATE employees status='離職', resign_date, resign_reason → 下個月薪資自動進入「離職結算」（含未休特休折現）
權限非 admin 只能看自己的申請（隱私保護）。admin 可看全部
取消申請中可由本人或 admin 點「取消」改 status='已取消'

D2. 留職停薪v2 新

路由Builder 自訂（DB 已建表，UI 待 builder 建）
DB 表leave_of_absence_requests
核准方式簽核鏈 category='留停'
欄位employee_id、start_date、planned_end_date、actual_return_date（實際回任日）、reason_type（產假/育嬰/兵役/進修/家庭/其他）、reason_detail、attachment_url
連動核准 → trg_loa_cascade trigger 自動 status='留停'。設定 actual_return_date 後 → 自動改回 '在職'
備註育嬰留停依性平法為法定權利，HR 不可拒

D3. 人事異動v2 新

路由/hr/forms/transfer
DB 表personnel_transfer_requests
核准方式簽核鏈 category='異動'
欄位 基本：employee_id、transfer_type（調職/升遷/降調/部門調動/跨店調動/調薪）、effective_date、reason
異動前 (snapshot)：old_department_id、old_store_id、old_position、old_base_salary、old_role
異動後：new_department_id、new_store_id、new_position、new_base_salary、new_role
連動最後一關核准 → DB trigger trg_transfer_cascade 自動：(a) 寫 position_history（保留歷史）；(b) UPDATE employees 主檔（含 base_salary）；(c) 下月薪資自動套新薪資數字
UI 特色表單顯示 DiffRow（前 → 後對比，劃掉舊值、紅字顯示新值）

D4. 試用期評核

路由/hr/probation

欄位employee（搜尋下拉）、mentor（指導人，搜尋下拉）、start_date、end_date（試用結束日，預設 90 天）、ratings（評分項目）、status（試用中/已通過/未通過/延長試用）、comments

H. 簽核鏈定義（共 4 條 HR 範本，今日新建）

進 [/process/approval-chains](/process/approval-chains) 點「✨ HR 範本」一鍵建立 4 條範本鏈：

鏈名category步驟（建立後請編輯填具體簽核人）
離職申請離職申請人 → 直屬主管 → HR 確認 → 執行長核准 → ✓
留停申請留停申請人 → 直屬主管 → HR 確認 → 執行長核准 → ✓
人事異動異動申請人 → 直屬主管 → HR 確認 → 執行長核准 → ✓
預先加班預先加班申請人 → 直屬主管 → 部門主管 → ✓

建立後預設 step.target_type='label'，每步只有標籤沒有指定簽核人。請逐條編輯、用搜尋下拉指派具體員工，否則送單會卡在「找不到第一關簽核人」。

I. 簽核流程通用機制

I1. 送單時（自動推第一審）

  1. 前端 INSERT 表單 record（含 approval_chain_id, current_step=0）
  2. 呼叫 RPC hr_chain_resolve_first_approvers(table, id) 找第一關合法簽核人
  3. 寫 notifications 表（recipient_emp_id 給每位簽核人）→ 出現在通知中心
  4. 未來：可加 LINE 推播

I2. 核准 / 駁回（hr_chain_approve RPC）

核准step+1 → 推下一關通知；最後一關 → status='已核准' + 觸發 DB trigger cascade
駁回status='已駁回'，記 reject_reason，員工可看到原因。current_step 保留（B2 模式：員工修改後從這關繼續）
取消申請中可由本人或 admin 取消，status='已取消'

I3. UI 顯示

  • 列表頁：每筆顯示「簽核進度」欄 → 各 step 用顏色區分（綠=已過、橘=當前、灰=等待）
  • 表單上方顯示「· 簽核鏈：XX（N 關）」或「⚠ 無簽核鏈」警示
  • 只有當前 step 的合法簽核人才看得到「核准 / 駁回」按鈕

M. 操作建議（給 admin）

  1. 第一步：進 [/process/approval-chains](/process/approval-chains) 點「✨ HR 範本」建立 4 條範本鏈

  3. 第三步：進 [/hr/forms](/hr/forms) 確認 4 大類顯示正確
  4. 第四步：用 admin 帳號送一筆假離職單測試流程是否從頭跑到尾

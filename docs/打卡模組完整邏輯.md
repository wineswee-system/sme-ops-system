# 打卡模組完整邏輯文件

> 版本：2026-05-29  
> 範圍：主系統（sme-ops-system）+ LIFF（sme-ops-liff）

---

## 一、五種打卡模式

| 模式鍵值 | 中文名稱 | 觸發條件 | 說明 |
|----------|----------|----------|------|
| `normal` | 一般 | 預設 | 全規則：位置驗證 + 時段驗證 |
| `overtime` | 加班 | 員工手動選 | bypass 時段限制；必須連結或自動建立加班申請 |
| `leave` | 請假 | 員工手動選 | 不扣遲到罰；必須在班別時段內；自動建立或連結請假申請 |
| `shift_swap` | 換班 | 員工手動選 | bypass 時段；**必須已有已核准的換班單**（不能臨建） |
| `outing` | 外出 | 員工手動選 | bypass 時段 + 位置驗證；自動建立或連結公出單 |

## 二、打卡資料欄位（`attendance_records`）

```
date              當日日期
clock_in          上班打卡時間（HH:MM）
clock_out         下班打卡時間（HH:MM）
clock_in_mode     上班打卡模式（5種之一）
clock_out_mode    下班打卡模式（5種之一，獨立設定）
status            當日狀態（正常 / 遲到 / 加班 / 請假 / 外出 / 補登）
is_late           是否遲到（boolean）
late_minutes      遲到分鐘數
total_hours       工作時數（下班 - 上班，跨午夜修正）
overtime_request_id  → overtime_requests.id（模式 overtime 時帶值）
leave_request_id     → leave_requests.id   （模式 leave 時帶值）
shift_swap_id        → shift_swaps.id       （模式 shift_swap 時帶值）
business_trip_id     → business_trips.id    （模式 outing 時帶值）
clock_in_method   gps / wifi / bypass
clock_in_lat/lng  打卡當下 GPS 座標
clock_in_ip       打卡 IP
```

**模式↔FK 一致性 CHECK（DB 層保護）**：
- `clock_in_mode = 'overtime'` → `overtime_request_id IS NOT NULL`
- `clock_in_mode = 'leave'` → `leave_request_id IS NOT NULL`
- `clock_in_mode = 'shift_swap'` → `shift_swap_id IS NOT NULL`
- `clock_in_mode = 'outing'` → `business_trip_id IS NOT NULL`
- `clock_in_mode = 'normal'` → 四個 FK 皆可為 NULL

## 三、上班打卡（clock_in）完整流程

```
員工按打卡 → 帶入參數：
  action = 'clock_in'
  clock_mode = normal | overtime | leave | shift_swap | outing
  lat, lng, accuracy（GPS 座標）
  ip（客戶端 IP）
  [選填] overtime_request_id / leave_request_id / shift_swap_id / business_trip_id
         （事先有申請單時可帶入直接連結，否則系統自動建立）
```

### 3.1 位置驗證（`outing` 模式跳過）

```
取得門市設定：lat/lng/clock_radius/allowed_wifi/clock_in_method

┌─ GPS 驗證 ─────────────────────────────────────────────────┐
│ haversine 距離 ≤ clock_radius（預設 200m）→ gpsPass = true  │
│ 精確度 > 200m → 直接失敗                                    │
└─────────────────────────────────────────────────────────────┘
┌─ WiFi 驗證 ─────────────────────────────────────────────────┐
│ clientIP 符合任一 allowed_wifi CIDR → wifiPass = true        │
└─────────────────────────────────────────────────────────────┘

clock_in_method = 'gps_required'  → 只接受 GPS，WiFi 不算
否則                               → GPS 或 WiFi 任一通過即可
全都失敗                           → 403 回傳失敗原因
```

### 3.2 模式各自的時段驗證與動作

#### `normal` 模式

```
呼叫 determineLateStatus()：

1. 查 schedules.actual_start（當日排班實際開始時間）
   ↓ 沒有
2. 查 schedules.shift → shift_definitions.start_time/end_time
   ↓ 沒有
3. fallback：stores.office_hours_start（預設 09:00）

early_clock_minutes（預設 30）：距上班超過 30 分鐘 → 拒絕「提早打卡」
late_tolerance_minutes（預設 5）：超過上班時間 5 分鐘以內 → 視為正常

結果：
  - 遲到 →  is_late = true, status = '遲到', late_minutes = 實際分鐘數
  - 正常 →  is_late = false, status = '正常'
  - 提早 →  拒絕打卡（回傳 403）
```

#### `overtime` 模式

```
1. 帶入 overtime_request_id？
   → 是：查表驗證（employee_id 相符 + date = 今日）
   → 否：自動 INSERT overtime_requests：
        status = '待審核', hours = 0, source = 'clock_in'
        reason = '打卡申請加班（上班 HH:MM，時數待確認）'

2. is_late = false（bypass 遲到判斷）
   status = '加班'

3. 打卡後 back-link：
   overtime_requests.attendance_record_id = 新建的 attendance_record.id
```

#### `leave` 模式

```
1. 仍呼叫 determineLateStatus()（確認時段合理性）：
   - 提早打卡 → 拒絕（同 normal）
   - 超過班別結束時間 → 拒絕（建議改用 overtime 模式）
   - 遲到 →「允許，但 status 改為 '請假'，不設 is_late = true」
   - 正常 → status 維持正常

2. 免遲到罰：is_late = false（無論幾點來都不扣）

3. 帶入 leave_request_id？
   → 是：驗證（employee_id + 日期範圍涵蓋今日）
   → 否：自動 INSERT leave_requests：
        type = '事假', status = '待審核', days = 1
        reason = '打卡申請請假調整（上班 HH:MM，請至 HR 補件）'
```

#### `shift_swap` 模式

```
1. 必須帶 shift_swap_id，否則拒絕（換班不能臨建，需走換班申請流程）

2. 驗證 shift_swaps：
   - status = '已核准'
   - swap_date = 今日
   - requester_id 或 target_id = 本員工

3. bypass 時段驗證
   is_late = false, status = '正常'
```

#### `outing` 模式

```
1. 已 bypass 位置驗證

2. 帶入 business_trip_id？
   → 是：驗證（employee 姓名相符 + 日期範圍涵蓋今日）
   → 否：自動 INSERT business_trips：
        status = '待審核'
        purpose = '打卡申請外出（上班 HH:MM）'

3. is_late = false, status = '外出'
```

## 四、下班打卡（clock_out）完整流程

```
前提：當日已有上班打卡且尚無下班打卡

1. 計算工作時數：
   workedMinutes = 下班時間 - 上班時間（支援跨午夜：負數 +1440）
   total_hours = workedMinutes / 60（四捨五入兩位）

2. 繼承上班打卡的 FK：
   overtime_request_id / leave_request_id / shift_swap_id / business_trip_id
   → 優先用 attendance_records 已有的值，若無才用本次帶入的 attachId
```

### 4.1 下班模式特殊邏輯

#### `normal` 模式

```
早退檢查（僅 normal 才做）：
  stores.office_hours_end 存在時，
  distance = office_end - 現在時間
  distance > late_tolerance_minutes → 拒絕「早退」
```

#### `overtime` 模式

```
1. 更新 overtime_requests：
   hours = 實際工作時數
   reason = '加班打卡（上班 HH:MM — 下班 HH:MM，時數 Xh）'

2. 若上班時未建 OT request（上班用 normal，下班改 overtime）：
   自動 INSERT overtime_requests（source = 'clock_out'）
   back-link attendance_record_id

3. status = '加班'
4. 不做早退檢查
```

#### `leave` 模式

```
1. 若辦公開始時間存在，確認現在 ≥ 辦公開始（避免異常早退）
2. 若無 leave_request_id，自動建立請假單
3. 不做早退檢查，不覆蓋上班時已設定的 status
```

#### `shift_swap` / `outing` 模式

```
不做早退檢查（bypass 時段規則）
```

## 五、補打卡（clock_corrections）流程

### 5.1 員工申請（LIFF）

```
欄位：
  date          補打卡日期
  type          上班打卡 / 下班打卡（對應 clock_in / clock_out）
  correction_time  補正的時間（HH:MM）
  clock_mode    normal / overtime / leave / shift_swap / outing
  reason        原因說明
  store         補打卡門市

RPC liff_insert_clock_correction：
  - 查 employee_line_accounts 找員工
  - 正規化 type：中文 → 英文（'上班打卡' → 'clock_in'）
  - 驗證 clock_mode（無效時 fallback 'normal'）
  - INSERT clock_corrections（status = '待審核'）
```

### 5.2 主管審核（LIFF 或 Web）

```
HR/主管審核 → 點「核准」→ 呼叫 liff_approve_request（或 web approve）
→ UPDATE clock_corrections SET status = '已核准'
→ DB Trigger 觸發（trg_clock_correction_apply）
```

### 5.3 核准後自動寫入出勤（DB Trigger）

```
Trigger：trg_clock_correction_apply
  AFTER UPDATE OF status ON clock_corrections
  FOR EACH ROW WHEN (NEW.status = '已核准')
  → 呼叫 _apply_correction_to_attendance(c)

函式邏輯：
  1. 跳過：correction_time IS NULL 或 type 不是 clock_in/clock_out

  2. 非 normal 模式 → find-or-create 對應申請單（滿足 FK CHECK constraint）：
     - overtime  → 在 overtime_requests 找同日同員工的單，找不到就自動建立
     - leave     → 在 leave_requests 找涵蓋當日的單，找不到就自動建立（類型：事假）
     - outing    → 在 business_trips 找，找不到就自動建立
     - shift_swap → 只找不建（需兩段確認流程）；
                   找不到就 fallback 成 normal，避免 CHECK 噴錯

  3. 查詢當日是否已有 attendance_record：
     - 有 → UPDATE（COALESCE 保護：不覆蓋另一端的值）
     - 無 → INSERT（status = '補登'）

  4. 更新欄位：
     clock_in  / clock_out           （依 type 決定更新哪個）
     clock_in_mode / clock_out_mode  （寫入 clock_mode）
     overtime_request_id / leave_request_id / shift_swap_id / business_trip_id
```

## 六、薪資計算如何使用打卡資料

主函式：`generate_payroll(p_pay_period)` — 每月跑一次

### 6.1 出勤工時（月薪制參考用）

```sql
SELECT COALESCE(SUM(total_hours), 0)
FROM attendance_records
WHERE employee_id = ? AND date BETWEEN 月初 AND 月末
-- 含所有模式，工時全算（外出也算工時）
```

### 6.2 加班費（overtime_requests）

```sql
SELECT SUM(ot_hours) 平日加班時數, SUM(ot_hours) 假日加班時數
FROM overtime_requests
WHERE employee_id = ? AND request_date BETWEEN 月初 AND 月末
  AND status = '已核准'
  AND (ot_type IS NULL OR ot_type = 'pay')
```

**注意**：打卡模式 `overtime` 會自動建立 `overtime_requests`（status = '待審核'），需要主管核准後才入薪資計算。

計費公式（勞基法 §24/§39）：
```
平日加班（v_ot_hours_wd）：
  ≤ 2h → × 時薪 × 1.34
  > 2h → 前2h × 1.34 + 超出部分 × 1.67

假日加班（v_ot_hours_hd）：
  前 2h → × 1.34
  3~8h  → × 1.67
  9~12h → × 2.67
```

### 6.3 換班（shift_swap）休息日 OT

```sql
-- 新邏輯（2026-05-29 起）
SELECT COALESCE(SUM(ar.total_hours), 0)
FROM attendance_records ar
WHERE ar.employee_id = ?
  AND ar.date BETWEEN 月初 AND 月末
  AND ar.shift_swap_id IS NOT NULL
  AND (DOW IN (0,6) OR 是國定假日)
  AND 同日無已核准 overtime_request
→ 加入 v_ot_hours_hd（假日 OT 計費）
```

換班若落在原本的休息日（週末/國定假日），視為休息日上班，自動納入假日 OT 計算。若 HR 另有建立 overtime_request，以 OT request 為準，不重複計算。

### 6.4 請假扣款

```sql
-- 全薪假：事假 / 無薪假
SELECT SUM(天數) FROM leave_requests
WHERE employee_id = ? AND status = '已核准'
  AND leave_type IN ('事假', '無薪假')
→ 扣款 = 天數 × 日薪

-- 半薪假：病假 / 生理假
SELECT SUM(天數) FROM leave_requests
WHERE leave_type IN ('病假', '生理假')
→ 扣款 = 天數 × 日薪 × 0.5
```

打卡模式 `leave` 自動建立的請假單預設為「事假」、status = '待審核'，**須 HR 核准後才扣款**。若要改為病假，HR 需在 leave_requests 修改 type。

### 6.5 遲到扣款（★ 2026-05-29 更新）

```sql
SELECT COALESCE(SUM(late_minutes), 0)
FROM attendance_records
WHERE employee_id = ? AND date BETWEEN 月初 AND 月末
  AND is_late = true
  AND COALESCE(clock_in_mode, 'normal') = 'normal'  -- 只算一般模式
```

扣款 = `FLOOR(遲到分鐘 ÷ 30) × 時薪 × 0.5`

**overtime / leave / shift_swap / outing 模式的打卡記錄不計遲到**。

### 6.6 全勤獎金判定

```
IF 遲到分鐘 > 0 OR 請假天數 > 0
  THEN 全勤獎金 = 0（扣光）
```

`outing` 模式（外出）不加到遲到或請假，全勤獎金**不受影響**。

## 八、申請單生命週期

### overtime_requests

```
[打卡申請路徑] clock_in edge function → status = '待審核', source = 'clock_in'
                clock_out edge function → hours 更新為實際工時, source = 'clock_out'

[補打卡路徑]   _apply_correction_to_attendance → 
               先查同日已有的 OT request → 無則建立（status = '待審核'）

[薪資納入條件] status = '已核准' + ot_type IS NULL OR ot_type = 'pay'

[注意] 打卡自動建立的 OT request hours = 0（上班時）；
       下班打卡後 hours 更新為實際時數；
       HR 需核准後才算加班費。
```

### leave_requests

```
[打卡申請路徑] clock_in edge function → type = '事假', status = '待審核'
                                        reason 提示「請至 HR 補件」

[補打卡路徑]   _apply_correction_to_attendance → 
               先查涵蓋當日的 leave request → 無則建立（事假，待審核）

[薪資扣款條件] status = '已核准'
               + leave_type IN ('事假', '無薪假') → 全薪扣
               + leave_type IN ('病假', '生理假') → 半薪扣

[HR 注意]  打卡自動建立的假單一律是「事假」，
           若員工實際是病假，HR 需手動修改 leave_type。
```

### business_trips

```
[打卡申請路徑] clock_in edge function → status = '待審核', purpose = '打卡申請外出...'
[補打卡路徑]   _apply_correction_to_attendance → 同上
[薪資影響]    目前無差旅費/外出津貼計算（需另行在 salary_structures 設定）
```

### shift_swaps

```
[不能臨建] 必須事先走換班申請流程（兩段確認）→ status = '已核准'
           打卡 / 補打卡 只能「查找」，找不到就 fallback 成 normal

[薪資影響]
  - 休息日（週末/國定假日）→ 自動計假日 OT
  - 平日互換 → 照常工時，無特殊處理
```

## 十、已知限制與 HR 注意事項

1. **打卡自動建立的申請單需人工核准**  
   overtime / leave / outing 打卡會建立「待審核」申請單，HR 需手動核准才進薪資。建議 HR 每日審核前一天的打卡申請單。

2. **leave 模式預設事假**  
   打卡時來不及說明請假原因，自動建「事假（全薪扣）」。若員工是病假，HR 必須在 leave_requests 修改 leave_type，否則扣款會錯。

3. **換班必須事先申請**  
   shift_swap 模式不能即時建立換班單，員工需前一天前完成換班申請流程並核准。

4. **補打卡不重算 is_late**  
   補打卡核准後寫入 attendance_records 時，不重新計算 is_late（沒有班表比對邏輯）。補登的出勤紀錄 status 固定為「補登」。

5. **外出（outing）目前無差旅費**  
   business_trip_id 有連結但薪資計算不加差旅費，如有需要需在 salary_structures.custom_allowances 手動設定。

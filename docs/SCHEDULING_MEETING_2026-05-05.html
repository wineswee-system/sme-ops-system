📌 一句話總結

雙引擎（程式 + AI）排班系統，支援週/月/變形工時三種視圖、班別制與時段覆蓋制兩種模式，自動套用勞基法 17 條硬性規則 + 10 條軟性建議，產出後可發布給員工 LINE 通知。

⚙️ 兩種引擎對照

📐 程式排班（Programmatic）

  • 速度：1–3 秒
  • 結果穩定：每次同輸入 = 同結果
  • 邏輯透明：每步驟都可追蹤
  • 免費：不用付 API 費用
  • 適合：日常排班、批次跑、無網路

🤖 AI 排班（Gemini）

  • 速度：10–30 秒（含重試）
  • 結果有變異：每次略有差異
  • 創意度高：能處理複雜偏好
  • 需付 API 費
  • 適合：複雜排班、偏好衝突多時

📅 三種視圖模式

視圖範圍適用工時制策略
週視圖7 天標準工時單週獨立排，最簡單
月視圖整個月月制工時目標切成 4–5 週逐週排，跨週累積工時
Cycle 視圖2 / 4 / 8 週變形工時以變形週期為單位，跨週累積工時與休假

🎯 兩種排班模式

📋 班別制 Shift-based

有預設班別（如「11–20 早班」「15–24 晚班」），員工指派整段班別。

  • 適合：固定營業時段、有明確班別
  • 每班別有最低人數要求
  • 支援平日/假日不同班別
  • 可區分 FT/PT 限定班別

🕐 時段覆蓋制 Time Slot

不限定固定班別，員工只需「覆蓋」需求時段（如 11–14 點需 2 人、19–22 點需 3 人）。

  • 適合：彈性排班、人力靈活運用
  • 系統自動算每人最佳起訖時間
  • 適合超商/早午餐等不固定班別
  • 啟用條件：門市有設定時段需求

🔁 排班核心流程

1排休假　→ 依員工類型算月休目標（FT 10 天 / PT 20 天），先放員工已申請的休
▼
2排班別 / 時段　→ 依排班優先序、員工偏好、最低人力，貪婪選最佳班別
▼
3修補開關店　→ 找有開店資格的員工頂替早班、有關店資格的頂替晚班
▼
4跨店借調　→ 本店缺人時從其他門市借調可支援員工
▼
5合規檢查　→ 套 17 條勞基法規則，違規標 error；10 條軟性規則標 warning
▼
6產出草稿　→ 顯示違規清單與工時統計，admin 審閱後發布

員工偏好（3 級設定）

優先（preferred）員工想要的班別 → 排班加分
中性（neutral）沒特別偏好 → 排班不加減分
避開（avoid）員工不想要的班別 → 排班減分

🔄 換班 / 代班流程

1A 員工想跟 B 換班 → 進「換班申請」選兩人的日期
▼
2B 員工收 LINE 通知 → 點同意 / 拒絕
▼
3B 同意後 → admin 進「換班申請」頁面審核
▼
4admin 核准 → 系統自動互換兩人班別
▼
5雙方收 LINE 通知排班已調整

🏪 跨店借調機制

本店人力不足時，從其他門市借調員工。每位員工可設定「我可支援的其他門市」。
  1. 系統找出本店缺人時段（人數 < 需求）
  2. 掃描所有可支援本店的他店員工
  3. 檢查該員工：當日未排其他店、通過勞基法、工時還有配額
  4. 排入班表，標記為「跨店班」
  5. 通知該員工本月有跨店班

🔗 與薪資的連動

實際工時累加為當月總工時 → 時薪員工底薪計算
班別類型休/特休/事假等 → 對應薪資（全薪/半薪/無薪）
是否「公司休」影響月休配額計算（員工請的假不算）
跨日班別（夜班）影響加班費級距判定
已批准請假排班會跳過該日，薪資依假別計算

📋 操作流程

步動作說明
1準備門市設定確認工時制、營業時間、人力規則、班別定義
2進入排班頁選門市、選月份/週/cycle
3員工排休員工自行用 LINE/手機提希望休 → 系統累積
4產生草稿點「AI 自動排班」或「程式排班」→ 等 1–30 秒
5審閱草稿看違規清單、調整不滿意的單格
6發布點「發布草稿」→ 員工 LINE 通知收到班表
7事後變更員工申請換班、補登、調休 → 走簽核流程

💡 開會討論重點

  1. 每位門市的工時制要確認　目前是設定在 store_settings — 標準工時 / 2 週 / 4 週 / 8 週變形 / 月制要逐店檢查
  2. 每天的開關店人員清單　哪些員工能開店、能關店要在員工檔逐個設定
  3. 跨店支援員工清單　哪些員工願意支援其他門市要設好
  4. 員工偏好如何收集　建議發給員工填一次（preferred / neutral / avoid），減少排班爭議
  5. 每月排班何時跑　建議 25 號跑下個月，留 5 天員工反饋與調整
  6. 違規硬性規則處理　目前出 error 會擋發布，要不要可以「強制覆寫 + 留註記」？

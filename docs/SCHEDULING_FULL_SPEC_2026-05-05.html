SME Ops System — 排班系統完整規格

產出 2026-05-05　·　核心：程式演算法 + AI 雙引擎，支援週/月/Cycle 三種視圖、班別制 + 時段覆蓋制兩種模式、勞基法 H1–H17 硬性約束

A. 系統定位

排班是 HR 模組最複雜的部分，涉及：

  • 勞基法合規：每日上限、連續工時、間隔、月加班上限、變形工時等 17 條硬性規則（H1–H17）
  • 營運需求：每時段必有開店/關店人員、最低人力、跨店支援
  • 員工偏好：希望休日、可上班時段、避開的班別
  • 數學最佳化：在以上限制下找滿意所有人的解（NP-hard 問題）

本系統採雙引擎策略：

引擎程式排班 (Programmatic)AI 排班 (Gemini)
速度1–3 秒10–30 秒（含 retry）
結果穩定每次相同輸入 = 同結果有變異性
邏輯可解釋步驟透明可 debug黑盒，需 retry 機制
適用常規排班、batch、無網路創意排班、複雜偏好
成本免費每次呼叫 API 費用

B. 三種視圖模式

模式範圍適用工時制排班策略
週視圖7 天（週一~週日）標準工時單週獨立排班
月視圖整個月（28–31 天）月制工時目標切成 4–5 週，跨週累積工時/休假
Cycle 視圖2 / 4 / 8 週變形工時以變形週期為單位（從 anchor date 起算），切週後跨週累積

store_settings.work_hour_system 決定 cycle 是 2 週 / 4 週 / 8 週。anchor 由 variable_period_start 設定（必須是週首日）。

C. 變形工時規則（勞基法 §30 / §30-1）

制度單週上限週期總時上限週期休息天數
標準工時40h—每 7 天 1 例假 + 1 休息日
2 週變形48h（單週）2 週 84h2 週 4 天
4 週變形48h（單週）4 週 168h4 週 8 天
8 週變形48h（單週）8 週 336h8 週 16 天（每 7 天 1 例假）
月制（自訂）—168h × (月份天數 / 28)每 7 天 1 例假；月制目標休假天數依員工類別

每日工作上限 §32：8h（一般）、12h（加班含原 8h）。

D. 兩種排班模式

D1. 班別制（Shift-based）— 預設模式

有預設班別（如 11–20 早班、15–24 晚班），員工指派整段班別。

欄位說明
shift_definitionsstore 的可用班別清單（name / start_time / end_time / break_minutes / employee_type / day_type）
store_staffing每班別所需人數（依平日/假日）
schedules.shift員工當日班別名稱（'11-20'、'休'、'特休' 等）

D2. 時段覆蓋制（Time Slot Coverage）— v2 新

不限定固定班別，員工只需「覆蓋」門市需求時段（如 11:00–14:00 需 2 人、19:00–22:00 需 3 人）。系統自動算每人最佳起訖時間。

欄位說明
store_time_slots每 store 一組時段 (start_time, end_time, required_count, day_type)
schedules.actual_start / actual_end系統自動算出的起訖時間（不再對應 shift_definitions）

啟用條件：store_time_slots 有資料 → useTimeSlotMode=true。否則用班別制。

F. 核心演算法（runProgrammaticSchedule 步驟）

輸入：data {employees, shiftDefs, timeSlots, weekDates, offRequests, preferences, storeSettings, ...} ↓ Step 0：解析 store_settings.work_hour_system → 取得 wsConstraints (週上限/週期總時/休假天數) ↓ Step 1：planRestDays · 依 employee_type 算月休目標 (FT 10 / PT 20) · 抓 off_requests 已申請的休 → 直接放 · 套用 weekly_target_hours 計算還需排幾天工作 ↓ Step 2：sortShifts by start_time（先排早班再排晚班） ↓ 分支判斷：useTimeSlotMode = (timeSlots.length > 0) ↓ ═══ IF useTimeSlotMode ═══ (時段覆蓋制) Step 3：對每天每員工： · 計算所需 grossH（粗工時 = 含休息時間） · 列舉所有可能的 [起時, grossH] 窗口 · 對每窗口算 score（時段覆蓋分 + 開關店分 + 工時偏好分 − 疲勞） · 選最佳窗口指派 ═══ ELSE ═══ (班別制) Step 3：兩段式班別指派 第一輪：員工依 schedule_priority 排序，貪婪選 score 最高的班別 第二輪：補齊未滿員班別（required_count 還有缺） Step 3b：開店/關店人員修補 · 找 can_open=true 的員工頂替 isOpening 班 · 找 can_close=true 的員工頂替 isClosing 班 Step 3c：跨店借調 (Cross-store) · 若仍缺人 → 從 additional_stores 包含本店的他店員工調入 ↓ Step 3b（共用）：補空缺 — FT 員工沒排到 → 給最早班別塞進去 ↓ Step 4：Build assignments 物件陣列 ↓ Step 5：validateResult — 套 H1-H17 + S1-S10 規則檢查 · 每筆違反 → 紀錄 violation { severity: 'error'/'warning', constraint, message } ↓ Step 6：computeStats — 算每員工本週工時 / 排班滿意度 / 疲勞度 ↓ 回傳 { assignments, violations, stats, reasoning, meta }

G. 月排班 / Cycle 排班的多週迴圈（runMonthlyProgrammaticSchedule）

輸入：data { monthDates, ..., storeSettings.work_hour_system } ↓ splitIntoWeeks(monthDates) → weeks[][] ↓ 若 cycle 模式： weekCycleIdx[i] = weeks[i] 屬於第幾個 cycle（用 anchor 推算） 跨 cycle 邊界的週標記為「歸屬多數天的 cycle」 ↓ allAssignments = [] monthHours, monthRestDays, monthFatigue = {} ↓ FOR i = 0 to weeks.length: IF cycle 切換（weekCycleIdx[i] 比上週大）→ reset accumulators weekData = { ...data, weekDates: weeks[i], previousWeek: 上週 assignments, monthlyContext: { hoursAccumulated, restDaysUsed, weeksRemaining: cycle 內剩幾週 } } result = runProgrammaticSchedule(weekData) ← 內部迴圈 allAssignments.push(...result.assignments) 累加 monthHours, monthRestDays, monthFatigue ↓ 最終校正：強制月休精確到目標天數（多排的休轉成班） ↓ validateMonthlyResult — 套 H6 (月加班 ≤46h)、H17 (月休 ≥X 天)、H11 (週期總時) ↓ 回傳 { assignments: allAssignments, violations, stats, monthFatigue }

H. 勞基法硬性約束（H1–H17 — 違反即為 error）

編號規則法源 / 說明
H1有 off_request 就不能被排班排班規則：員工請假已批准的日子，被排成上班 = 違反
H2每日工時 ≤ 12 小時勞基法 §32：含加班最多 12h
H3連續工作天數 ≤ 12（FT）/ 6（PT）勞基法 §36：每 7 天 1 例假
H4跨日工時間隔 ≥ 11 小時勞基法 §34：輪班制 11h 休息
H6月加班 ≤ 46h勞基法 §32：含變形 54h
H7每週工時不超過 wsConstraints.weeklyMax + buffer動態依工時制計算
H8每週至少 1 例假勞基法 §36
H9can_open / can_close 員工排班限制非開店員工不得排早於 9 點開店班
H10未成年員工不得排夜班勞基法 §48
H11變形工時週期總時不超過 168h（4 週）/336h（8 週）勞基法 §30-1
H13孕婦/哺乳員工不得排夜班性平法 §15、勞基法 §49
H14員工不能排到非自己門市的班除非該店在 employees.additional_stores 內
H15員工類型須與班別 employee_type 匹配FT 不能排 PT-only 班；反之亦然
H16day_type 須與該日類型匹配weekday-only 班別不可排假日；反之亦然
H17月休天數須符合 store_settings 設定FT ≥10 天、PT ≤20 天

I. 軟性約束（S1–S10 — 違反為 warning）

編號規則說明
S1每天每班別人數 ≥ store_staffing.required_count低於需求 = warning（不阻擋發布）
S5月工時介於 monthlyMin / monthlyMax正職 150–175h、PT 80–175h（依日數比例）
S8每天有開店資格者 + 關店資格者否則上班/打烊無人交接
S9連續週末工作 ≤ 2 週公平性原則
S10時段覆蓋制：每時段覆蓋人數 ≥ required_countv2 新

J. 員工屬性詳解（影響排班決策）

欄位說明
employment_type'正職' / '兼職' / 'PT' — 影響月休目標、工時上限、可排班別
can_open是否能開店（開貨、開機台）— 影響早班指派
can_close是否能關店（盤點、結帳、上鎖）— 影響晚班指派
additional_stores可支援的其他 store_id 陣列 — 跨店借調基礎
weekly_target_hours該員工每週目標工時（FT 預設 40，可個別調整）
personal_hour_cap個人月工時上限（如孕婦不得超過 160h）
schedule_priority排班優先序（數字小者優先排）— 用於有衝突時的 tiebreak
is_pregnant / is_nursing觸發 H13（不得排夜班）
store_id主要門市 — 班別會優先排在此店

K. 與薪資的連動

排班端薪資端如何使用
schedules.actual_hoursgenerate_payroll 累加為當月總工時 → 時薪員工底薪計算
shift 類型（休/特休/事假...）查 ABSENCE_CONFIG payRate → 算扣薪
休假是否「公司休」countsAsRest=true 才吃月休配額；員工請的假另計
跨日班別（夜班）影響加班費級距判定（平日 vs 例假）
off_requests 已批准排班會跳過該日；薪資依假別 type 算

L. 換班 / 代班流程（shift_swaps）

A 員工想跟 B 換班 ↓ A 進「換班申請」→ 選 original_date (我的班)、target (B)、target_date (B 的班) ↓ INSERT shift_swaps (status='待 B 同意') ↓ B 收 LINE 通知 → 點同意 / 拒絕 ↓ 若 B 同意 → status='待 admin 核可' ↓ admin 進「換班申請」頁面 → 點核准 ↓ update schedules：A↔B 互換 shift / actual_start / actual_end ↓ 推 LINE 通知雙方排班已調整

M. 跨店調度（Cross-store Borrowing）

當本店人力不足，從其他門市借調員工的機制。

  1. 找出本店缺人時段（current 員工數 < required_count）
  2. 掃 employees WHERE additional_stores 包含本店 ID（且非本店 store_id）
  3. 檢查該員工：(a) 該日未排其他店；(b) 通過 H1–H17 法規；(c) 週/月工時還有配額
  4. 排入 schedules，標記 is_cross_store=true、home_store=原店
  5. 通知該員工本月有跨店班

N. 操作流程

  1. 準備：確認門市設定（工時制、營業時間、人力規則）→ 確認班別定義或時段覆蓋設好
  2. 進排班頁 [/hr/schedule]：選門市、選月份/週/cycle
  3. 員工排休：員工自行進 LIFF 提希望休 → off_requests 累積
  4. 產生草稿：點「AI 自動排班」(Gemini) 或「排班代碼」(程式) → 等 1–30 秒
  5. 審閱草稿：上方顯示 「程式排班 V2 / X 位員工 / Y 天」與是否合規。下方表格逐人逐日
  6. 調整：手動拉動單格修改、或重跑演算法
  7. 發布：點「發布草稿」→ schedule_publish_status 寫入、員工 LINE 通知
  8. 事後變更：員工申請換班、調休、補登 → 走簽核流程
